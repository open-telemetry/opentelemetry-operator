# TRACING-5835: Instrument description conflict for jvm_memory_used_bytes

## Summary

When a Java application (especially Spring Boot) is auto-instrumented by the OTel Java
agent, the collector's Prometheus exporter logs an info-level warning:

```
Instrument description conflict, using existing
  instrument: jvm_memory_used_bytes
  existing:   "The amount of used memory"
  dropped:    "Measure of memory used."
```

The metric still works (the exporter keeps the first description), but the warning is
noisy and indicates a real mismatch.

## Root Cause

Two independent sources emit `jvm.memory.used` with different descriptions:

| Source | Description | Origin |
|--------|-------------|--------|
| **Micrometer** (`JvmMemoryMetrics`) | "The amount of used memory" | Spring Boot / Micrometer standard |
| **OTel semantic conventions** | "Measure of memory used." | [jvm-metrics.md](https://github.com/open-telemetry/semantic-conventions/blob/main/docs/runtime/jvm-metrics.md#metric-jvmmemoryused) |

The OTel Java agent has a **Micrometer bridge** that forwards Micrometer-registered metrics
into the OTel pipeline. When a Spring Boot app has Micrometer's `JvmMemoryMetrics` active,
the bridge forwards `jvm.memory.used` with Micrometer's description alongside the agent's
own JVM instrumentation using OTel semantic convention descriptions. Both arrive at the
collector with the same metric name but different descriptions. The Prometheus exporter
keeps whichever it saw first and logs the conflict per the
[OTel spec](https://github.com/open-telemetry/opentelemetry-specification/pull/2890).

This is **not limited to `jvm.memory.used`** -- any JVM metric that both Micrometer and the
OTel agent emit will conflict if their descriptions don't match.

## Environment

- OCP 4.18
- Collector v0.135.0
- Java agent v1.33.6 (the operator's default -- see below)

## The Two Java Agent Versions in the Operator

The operator tracks two separate Java agent versions:

| File | Value | Purpose |
|------|-------|---------|
| `versions.txt` | `autoinstrumentation-java=1.33.6` | **Default image tag** the operator injects (the v1.x line, pinned intentionally -- comment says "should not be greater than 1.x.x") |
| `autoinstrumentation/java/version.txt` | `2.26.1` | Version used to **build** the published v2.x image |

The operator publishes both v1.x and v2.x images, but defaults to v1.x for backwards
compatibility. Multiple PRs to bump `versions.txt` to v2.x have been
[closed without merging](https://github.com/open-telemetry/opentelemetry-operator/pull/2631).

## The Fix Already Exists Upstream (in v2.0.0)

The Micrometer bridge was **disabled by default** in Java agent v2.0.0 (Jan 2024) via
[PR #10038](https://github.com/open-telemetry/opentelemetry-java-instrumentation/pull/10038),
which resolved [issue #9762](https://github.com/open-telemetry/opentelemetry-java-instrumentation/issues/9762).
The rationale was exactly this bug:

> "Micrometer bridge duplicates a lot of metrics that are generated by the javaagent itself.
> The micrometer metrics also don't implement the semantic conventions (yet)."

Users on **v2.x do not hit this bug** (unless they explicitly re-enable the bridge).
Users on **v1.x hit it by default** whenever the instrumented app uses Micrometer.

## Workaround for v1.x

On v1.x, the Micrometer bridge can be disabled via the standard instrumentation toggle:

```
OTEL_INSTRUMENTATION_MICROMETER_ENABLED=false
```

This can be set in the `Instrumentation` CR's `spec.java.env` field.

## Recommended Fix

The fix is upstream in the OTel Java agent. No operator-side code change is needed.

The options, in order of preference:

1. **Short term: document the workaround.** Users on the default v1.x agent can set
   `OTEL_INSTRUMENTATION_MICROMETER_ENABLED=false` in their `Instrumentation` CR.

2. **Medium term: bump the operator default to v2.x.** This would fix this bug and many
   others, but it's a **breaking change** (metric names, units, and attributes all changed
   in v2.0). The operator maintainers have intentionally avoided this so far. This is a
   conversation for the upstream operator community, not something to force.

3. **No-op alternative: upstream backport to v1.x.** The v1.x line is effectively EOL
   (v1.33.6 from Aug 2024 was the last release). A backport is not realistic.
